## How the achievement flow works
- Client flow (existing behavior): `addBeer` inserts into `beers`, fetches recent beers and lifetime counts, calls `checkAchievements` (client) and inserts any new badges into `achievements`.
- Server flow (new, preferred): A DB trigger (`AFTER INSERT ON public.beers`) runs `public.award_achievements()` which computes badges and inserts missing rows into `public.achievements`. This makes awarding automatic and atomic.

## Caveats and notes
- The DB trigger logic relies on `created_at` on inserted rows; client may set `created_at` explicitly or database default will be used.
- We added `api` views and triggers to allow PostgREST to route REST requests while keeping the underlying `public` tables protected.
- Migrations were applied using the Supabase CLI from this workspace; confirm these migrations are in your repository and source-controlled.
